# SPEC.md — Open-Source Python Baseball Pitch Physics Simulator

## 1. Project Summary

Build an educational open-source Python package for modeling and simulating baseball pitch physics, with a focus on the **Magnus effect**, drag, spin axis, active/gyro spin, seam effects, and pitch movement metrics.

The package should let users answer questions like:

* Why does a four-seam fastball appear to “rise”?
* Why does a curveball drop more than a spinless pitch?
* How do velocity, spin rate, spin axis, and spin efficiency change pitch movement?
* Why does gyro spin not create much movement?
* Why can two pitches with similar velocity and spin move differently?
* How do drag, air density, altitude, and wind affect a pitch?
* Why are seam-shifted wake effects hard to predict from simple Magnus-only models?

The package should be accurate enough for education, experimentation, and analytics prototyping, while being explicit that it is **not** a high-fidelity CFD solver or professional pitch-design system.

## 2. Recommendation

### Build this as three layers

1. **Core Python physics engine**

   * Pure simulation logic.
   * No Streamlit dependency.
   * No plotting dependency beyond optional Matplotlib helpers.
   * Uses SI units internally.
   * Exposes clean dataclasses and simulation outputs.

2. **Visualization layer**

   * 2D plots by default.
   * Optional 3D interactive Plotly visualizations.
   * Pitch comparison, force vectors, spin-axis arrows, catcher view, side view.

3. **Streamlit educational app**

   * Optional demo app, not the core package.
   * Sliders for pitch speed, spin rate, spin axis, spin efficiency, drag model, altitude, wind, and seam-shifted wake toy force.
   * Designed for learning and sharing, not required for library use.

This keeps the package useful for notebooks, scripts, and serious users, while the Streamlit app becomes the approachable educational layer.

## 3. Why This Package Should Exist

Existing baseball Python tools are mostly data-oriented. For example, `pybaseball` retrieves Statcast, Baseball Savant, FanGraphs, and Baseball Reference data, including pitch-level data, but it is not primarily a physics simulator. ([PyPI][1])

This package should fill a different niche:

> **An educational physics simulator for baseball pitch trajectories, built around transparent equations, tunable aerodynamic models, and interactive visual intuition.**

It should not compete with `pybaseball`; it should optionally integrate with it.

## 4. Important Physics to Understand

### 4.1 Gravity

Gravity is always present and pulls the ball downward. A “rising fastball” does not literally rise in normal game conditions; it generally **drops less than a comparable pitch with less upward lift**. Statcast’s induced vertical break explicitly removes gravity to isolate movement generated by how the pitcher spins and manipulates the ball. ([MLB.com][2])

### 4.2 Drag

Drag acts opposite the ball’s velocity relative to the air. It slows the ball from release to home plate and changes how much time gravity and lift have to act.

The drag force should be modeled as:

```text
F_D = -0.5 * rho * C_D * A * |v_rel| * v_rel
```

Where:

* `rho` = air density
* `C_D` = drag coefficient
* `A` = cross-sectional area
* `v_rel` = ball velocity relative to wind

Important: `C_D` is not truly constant. Baseball drag depends on Reynolds number, spin, seam geometry, surface roughness, and ball-to-ball variation. Baseball aerodynamics is complicated because the stitches disturb the boundary layer and the flow is separated, unsteady, and non-uniform. ([NASA][3])

### 4.3 Magnus Lift

A spinning baseball experiences an aerodynamic force perpendicular to both its velocity and spin axis. Nathan et al. experimentally measured spinning baseballs over speeds of 50–110 mph, spin rates of 1500–4500 rpm, and spin factors `S = Rω/v`; they found the lift coefficient depends strongly on spin factor and not strongly on speed at fixed `S`. ([ADS][4])

The simulator should model Magnus force as:

```text
S = R * |omega_perp| / |v_rel|

F_M = 0.5 * rho * C_L(S, Re, seam_model) * A * |v_rel|^2 * unit(omega x v_rel)
```

Where:

* `omega` = angular velocity vector
* `omega_perp` = spin component perpendicular to the velocity vector
* `S` = spin factor / spin parameter
* `C_L` = lift coefficient
* `unit(omega x v_rel)` = direction of Magnus force

Backspin creates upward lift. Topspin creates downward lift. Sidespin creates lateral movement. Gyro spin, where spin is mostly aligned with velocity, contributes little to Magnus movement.

### 4.4 Active Spin vs. Gyro Spin

Not all spin creates movement. MLB Statcast calls the portion of spin that contributes to movement **Active Spin**; spin that behaves more like a football or bullet is inactive/gyro spin. ([MLB.com][5])

This should be a first-class concept in the package.

```text
active_spin_fraction = |omega_perp| / |omega|
gyro_spin_fraction = |dot(omega, v_hat)| / |omega|
```

The package should let users specify either:

* full 3D spin vector, or
* spin rate + spin axis + spin efficiency / active spin percentage.

### 4.5 Spin Axis, Spin Direction, and Coordinate Confusion

Spin axis is easy to confuse with movement direction. Barton Smith notes that spin axis, spin direction, tilt, pitcher view, and catcher view are often mixed up, and that Magnus movement is 90 degrees from the spin axis. ([Baseball Aerodynamics][6])

The package should make this explicit:

* Use one internal coordinate convention.
* Clearly document pitcher-view vs. catcher-view.
* Always show the spin-axis vector and Magnus-force vector.
* Avoid ambiguous “tilt” labels unless the convention is defined.

### 4.6 Seam Effects and Seam-Shifted Wake

A Magnus-only model is educationally useful but incomplete. Seam orientation can change the wake and create movement not explained by spin alone. Barton Smith summarizes the core point bluntly: **seams can cause two pitches with identical initial trajectory, spin, and axis to go different places**. ([Baseball Aerodynamics][7])

Lyu et al. also found that lift and drag depend on spin, seam orientation, and flow separation; two-seam and four-seam orientations can produce different lift behavior, especially at low spin factor, and the flow is complex enough that simplified models have real limits. ([baseball.physics.illinois.edu][8])

Recommendation: include seam-shifted wake as an **optional empirical/non-Magnus force model**, not as a deterministic CFD-grade seam solver in v1.

## 6. Core Design Principles

1. **Educational first**

   * Equations should be visible.
   * Assumptions should be documented.
   * Users should be able to turn forces on/off.

2. **SI units internally**

   * Store meters, seconds, kg, radians.
   * Provide helpers for mph, rpm, feet, inches.
   * Avoid silent unit ambiguity.

3. **Composable force models**

   * Gravity model.
   * Drag model.
   * Magnus model.
   * Optional seam-shifted wake model.
   * Optional wind/environment model.

4. **Transparent empirical models**

   * Allow constant `C_D` and `C_L`.
   * Allow empirical `C_L(S)` models.
   * Allow user-defined coefficient functions.
   * Allow lookup-table or fitted models later.

5. **Validation over complexity**

   * Validate against analytic cases.
   * Validate qualitatively against expected pitch behavior.
   * Optionally compare to Statcast-style movement metrics.

6. **No overclaiming**

   * The package should not claim to exactly reproduce MLB pitches without full release, spin-axis, seam-orientation, environmental, and ball-property data.

## 7. Coordinate System

Use a right-handed coordinate system:

```text
x = horizontal, positive to catcher’s right
y = forward, from release point toward home plate
z = vertical, positive upward
```

At release:

```text
position = [x0, y0, z0]
velocity = [vx0, vy0, vz0]
spin = [omega_x, omega_y, omega_z]
```

Plate crossing occurs when:

```text
y = plate_distance
```

Default:

```text
release_to_plate_distance = 55 ft
```

This should be configurable because actual release distance depends on extension.

## 8. Physics Model

### 8.1 State Vector

```python
state = [
    x, y, z,
    vx, vy, vz,
]
```

Spin can initially be treated as constant over the short pitch flight:

```python
omega = constant
```

Future versions may include spin decay, but v1 should avoid unnecessary complexity.

### 8.2 Equation of Motion

```text
d(position)/dt = velocity

m * d(velocity)/dt =
    F_gravity
  + F_drag
  + F_magnus
  + F_non_magnus_optional
```

### 8.3 Gravity

```text
F_g = [0, 0, -m*g]
```

### 8.4 Drag

```text
F_D = -0.5 * rho * C_D(Re, S, seam) * A * |v_rel| * v_rel
```

### 8.5 Magnus

```text
omega_perp = omega - dot(omega, v_hat) * v_hat
S = R * |omega_perp| / |v_rel|

F_M = 0.5 * rho * C_L(S, Re, seam) * A * |v_rel|^2 * unit(omega x v_rel)
```

### 8.6 Optional Non-Magnus / Seam-Shifted Wake Term

Start with a simple empirical model:

```text
F_SSW = 0.5 * rho * C_SSW * A * |v_rel|^2 * direction_vector
```

Where `C_SSW` and `direction_vector` are user-specified or calibrated.

This is intentionally humble. Seam-shifted wake depends on seam orientation, boundary-layer separation, spin axis, and ball orientation. A deterministic seam model should be a later research feature, not an MVP promise.

## 10. Simulation API

### 10.1 Basic Usage

```python
from pitchphys import PitchRelease, Environment, simulate
from pitchphys.units import mph, rpm, ft

pitch = PitchRelease.from_mph_rpm_axis(
    speed_mph=95,
    spin_rpm=2400,
    spin_axis_deg=180,
    active_spin_fraction=0.95,
    release_height_ft=6.0,
    release_side_ft=-1.5,
    extension_ft=6.0,
)

env = Environment.sea_level()

traj = simulate(pitch, env=env, model="magnus")

print(traj.plate_location())
print(traj.break_metrics())
```

### 10.2 Compare Pitches

```python
fastball = PitchRelease.four_seam(speed_mph=96, spin_rpm=2500, ivb_axis="backspin")
curveball = PitchRelease.curveball(speed_mph=81, spin_rpm=2800)
slider = PitchRelease.gyro_slider(speed_mph=87, spin_rpm=2500, active_spin_fraction=0.25)

results = simulate_many([fastball, curveball, slider])
```

### 10.3 Force Toggles

```python
simulate(pitch, forces=["gravity"])
simulate(pitch, forces=["gravity", "drag"])
simulate(pitch, forces=["gravity", "drag", "magnus"])
simulate(pitch, forces=["gravity", "drag", "magnus", "seam_shifted_wake"])
```

This is essential educationally because users can see how each force changes the trajectory.

## 14. Testing and Validation

### 14.1 Unit Tests

Required tests:

* Gravity-only trajectory matches analytic projectile motion.
* No forces means constant velocity.
* Drag always opposes relative velocity.
* Backspin creates upward Magnus force.
* Topspin creates downward Magnus force.
* Pure gyro spin creates near-zero Magnus force.
* Higher active spin creates more movement, all else equal.
* Plate crossing interpolation is stable.
* Unit conversions are correct.

### 14.2 Scientific Sanity Checks

Examples:

* Four-seam fastball: positive induced vertical break.
* Curveball: negative induced vertical break.
* Gyro slider: high raw spin but relatively low movement.
* Same speed/spin but different spin axis produces different break direction.
* Gravity-only pitch drops more than a backspin fastball.

### 14.3 Data Validation

Optional validation against public Statcast-style data:

* Import real pitch parameters.
* Simulate a Magnus-only expected movement.
* Compare to observed horizontal/vertical movement.
* Attribute residual movement to model error, seam effects, measurement noise, or missing parameters.

This is a good educational feature because it shows both the power and the limits of simplified models.

> This package teaches and simulates the dominant first-order physics of pitched baseball flight, while exposing where simplified models break down.

You probably **do not need accelerated compute for v0.1**. I’d design for it, but not require it.

My recommendation:

> **Core MVP: NumPy + SciPy.
> Optional acceleration later: Numba first, JAX second.
> Avoid GPU dependency unless you are doing massive parameter sweeps, optimization, or differentiable fitting.**

## Recommended architecture

### 1. Default backend: NumPy + SciPy

Use this for the core educational package.

Why:

* A baseball pitch is a tiny ODE problem: 3D position + 3D velocity over roughly 0.4–0.5 seconds.
* One pitch simulation is cheap.
* The hard part is model clarity, coordinate conventions, force definitions, units, and educational visualization — not raw compute.
* SciPy’s `solve_ivp` is the right baseline for an initial-value ODE solver. It supports event handling, which is useful for stopping the simulation when the ball crosses home plate. SciPy’s own docs note that `vectorized=True` is mainly useful for finite-difference Jacobian estimation in stiff solvers and can be slower for small problems in some circumstances, so I would not over-focus on SciPy-level vectorization for the MVP. ([SciPy Documentation][1])

Use:

```python
numpy
scipy.integrate.solve_ivp
```

Good for:

* one pitch
* educational notebooks
* Streamlit sliders
* comparing a few pitch types
* correctness-first development

---

### 2. Vectorized NumPy: yes, but mainly for batch sweeps

Use vectorized NumPy for running many pitches at once, not because one pitch needs it.

Good examples:

* sweep spin rate from 1500–3500 rpm
* sweep spin efficiency from 0–100%
* sweep spin axis angles
* compare altitude / air density assumptions
* generate heatmaps of break vs. velocity/spin

For this, I’d eventually add:

```python
simulate_batch(pitches: PitchBatch, backend="numpy")
```

But I would not force every simulation into a fully vectorized shape immediately. That can make the code harder to teach.

Educational code should prefer clarity first:

```python
force = gravity + drag + magnus
```

Then add batch APIs once the single-pitch model is validated.

---

### 3. Numba: best optional acceleration for v0.2/v0.3

If you need speed, I would add **Numba before JAX**.

Why:

* It works well for numerical loops over NumPy arrays.
* It lets you keep a mostly normal Python/NumPy implementation.
* It is useful for fixed-step integrators and parameter sweeps.
* It has less conceptual overhead than JAX for typical scientific Python users.

Numba’s core value proposition is JIT-compiling Python/NumPy-style numerical code into fast machine code. ([Numba][2])

Recommended use:

```python
pip install pitchphys[numba]
```

Add a backend:

```python
simulate_batch(..., backend="numba")
```

Best use case:

```text
10,000 to 10,000,000 pitch simulations for sweeps, heatmaps, Monte Carlo uncertainty, or parameter fitting.
```

I would use Numba for a simple fixed-step RK4 or semi-implicit integrator, not necessarily around `solve_ivp`.

Example backend design:

```python
class IntegratorBackend:
    def simulate_one(...)
    def simulate_batch(...)
```

Backends:

```text
"scipy"  -> accurate, flexible, event-based, default
"numpy"  -> simple fixed-step vectorized batch
"numba"  -> fast fixed-step batch
"jax"    -> differentiable / accelerator-oriented batch
```

---

### 4. JAX: powerful, but not the default

JAX is attractive, but I would not make it a core dependency.

Use JAX only if the project grows toward:

* differentiable pitch physics
* fitting aerodynamic coefficients from data
* gradient-based optimization
* large batched simulations
* inverse modeling: “what spin axis / lift coefficient explains this observed movement?”
* GPU/TPU acceleration
* automatic vectorized sweeps

JAX gives you transformations like `jit`, `vmap`, and `grad`; `vmap` is designed for automatic batching/vectorization, and JAX’s docs describe it as a NumPy-like numerical system with automatic differentiation and JIT compilation. ([JAX Docs][3])

But the tradeoffs are real:

* JAX wants more functional, array-oriented code.
* It can complicate object-oriented dataclasses.
* It introduces installation complexity.
* It may be overkill for education-first users.
* Debugging JAX-traced code is less beginner-friendly.
* Streamlit interactivity probably will not need it.

So I’d make it optional:

```bash
pip install pitchphys[jax]
```

And frame it as advanced:

```python
simulate_batch(..., backend="jax")
fit_aero_coefficients(..., backend="jax")
```

## Bottom line

Do **not** start with JAX or GPU acceleration. Start with:

```text
NumPy + SciPy core
clean force models
excellent tests
2D/3D visualization
Streamlit educational demo
```

Then add:

```text
Numba for fast batch simulation
JAX for differentiable inverse modeling and optimization
```

That gives you the best tradeoff: simple enough for students and baseball fans, extensible enough for serious technical users.
